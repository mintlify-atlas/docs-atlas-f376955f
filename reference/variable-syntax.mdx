---
title: "Variable syntax"
description: "How to write dynamic variable placeholders in machote templates so they become required form fields during document generation."
---

When you write a machote template, you can embed variable placeholders directly in the HTML content. Each placeholder is extracted by the system and turned into a required input field on the document generation form.

## Placeholder format

The canonical syntax uses **square brackets**:

```
[VariableName]
```

Spaces immediately inside the brackets are trimmed, so `[ VariableName ]` is treated identically to `[VariableName]`.

Variable names are **case-sensitive**. `[NombreEmpleado]` and `[nombreempleado]` are treated as two different variables.

## Common variable examples

| Placeholder | Generated field label |
|-------------|----------------------|
| `[NombreEmpleado]` | NombreEmpleado |
| `[FechaInicio]` | FechaInicio |
| `[PuestoTrabajo]` | PuestoTrabajo |
| `[Salario]` | Salario |
| `[NumeroExpediente]` | NumeroExpediente |

Every extracted variable is created with `required: true` and `type: "string"`. There is currently no way to mark a variable as optional or change its input type from the template itself.

The following HTML snippet shows a constancia laboral with five variables. All five will appear as mandatory fields before the document can be generated.

```html Template HTML

  <strong>[NombreEmpleado]</strong>, con número de expediente

  salario mensual de <strong>[Salario]</strong>, a partir del
```

## Extraction regex

The system extracts variables using the following regular expression, defined in `src/app/services/api.ts`:

```typescript
const regex = /\[\s*([^\]]+?)\s*\]/g;
```

This regex matches any text enclosed in square brackets, capturing the trimmed inner content as the variable key. The match runs over the full HTML string of the template.

<Warning>
  If you use the same variable name more than once in a template, it still creates only **one** form field. The extraction function deduplicates using a `Set`, so repeated placeholders like `[NombreEmpleado]` appearing three times in a document will produce a single "NombreEmpleado" input. The same value fills every occurrence when the document is rendered.
</Warning>

## Legacy `{{Variable}}` syntax

If you paste content generated by an AI tool or an older template that uses double curly braces, the system will automatically convert it to square-bracket syntax before saving.

The `normalizePlaceholders()` function in `src/app/services/api.ts` applies this conversion:

```typescript
// Converts {{Variable}} → [Variable]
let normalized = text.replace(/\{\{\s*([^}]+?)\s*\}\}/g, '[$1]');

// Also converts bare {Variable} (uppercase-starting only, outside CSS/JSON)
normalized = normalized.replace(
  /(?<![{"\'])\{([A-Z][A-Za-z0-9_]*)\}(?![}"\'])/g,
  '[$1]'
);
```

<Note>
  You do not need to manually convert old templates. Paste them into the editor and the normalization runs automatically. However, writing new templates with `[Variable]` syntax from the start avoids any ambiguity with CSS `{}` blocks in the HTML.
</Note>

## How variables become form fields

The full lifecycle from template to filled document is:

1. You write HTML with `[VariableName]` placeholders in the machote editor.
2. On save, `extractVariablesFromSquareBrackets()` scans the HTML and produces the `variables` array stored on the machote in MongoDB.
3. When a user opens the machote to generate a document, the system reads the `variables` array and renders one text input per entry.
4. The user fills in each field. All fields are required — the form cannot be submitted until every variable has a value.
5. The system replaces each `[VariableName]` occurrence in the HTML with the submitted value and renders the final document.
